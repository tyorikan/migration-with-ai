# 6. テスト (Testing)

本セクションでは、モダナイズしたアプリケーションが正しく動作するかを保証するための、テスト戦略と自動化の手法を学びます。Gemini を活用してテストコードを生成し、CI パイプライン内で自動実行する基盤を構築します。

## ゴール
1. **AI 駆動テスト(AI-Driven Testing):** Gemini を使ってボイラープレートとなるテストコード（単体テスト、統合テスト等）を自動生成する。
2. **テスト自動化基盤の理解:** CI 環境 (Cloud Build や GitHub Actions など) でテストを自動実行する仕組みを構築できる状態にする。
3. **品質ゲートの担保:** コードをデプロイする前に、最低限の品質基準（テストカバレッジや成功率）をクリアさせるフローを体験する。

## テスト戦略の全体像

モダナイゼーションにおけるテストは、元の仕様（SFDC側）と同様の振る舞いをするか（リグレッション・テスト）が重要になります。

1.  **単体テスト (Unit Tests):** 個々の関数やメソッドが期待通りに動作するかを検証します。Geminiが最も得意とする領域です。
2.  **統合テスト (Integration Tests):** データベース (Cloud SQL/AlloyDB 上の PostgreSQL など) や外部 API との連携が正しく行われるかを検証します。ここではテスト用のデータベース（インメモリや Testcontainers を用いた PostgreSQL コンテナ）を利用することが推奨されます。特に、SFDC特有の挙動を模倣したデータのCRUD操作を検証します。
3.  **E2E テスト (End-to-End Tests):** API の呼び出しからレスポンスまで、全体を通した動作を確認します。

## Gemini を用いたテストコード生成のテクニック

Gemini にテストコードを書かせる際は、以下のコンテキストを与えると精度が高まります。

1.  **テスト対象のコード:** もちろん必須です。
2.  **使用するテストフレームワーク:** (例: `Go の testing パッケージと testify/assert を使って`, `Jest を使って`)
3.  **テストデータの例:** (例: `SFDCのアカウント情報を模したJSONを含むテストデータを作成して`)
4.  **網羅すべきエッジケースの指示:** (例: `正常系だけでなく、DB接続エラー時や、不正な入力値が来たときの異常系（境界値テストを含む）のテストケースも生成して`)

## ワークショップでのステップ

### Step 1: 単体テストの自動生成
1. 「3-app-modernize」で作成したビジネスロジックのファイル（例: `main.go` または `handlers.go`）を開きます。
2. Gemini に上記のプロンプト例のような指示を出し、テストコード (`*_test.go`) を生成させます。
3. 生成されたテストコードをコピーしてプロジェクト内に配置し、実行してみます（必要に応じて手動で微調整を行います）。

### Step 2: テスト実行スクリプトの作成
1. ローカル、または Cloud Shell などの CI 環境でテストを実行・評価するためのスクリプトを用意します。
2. `scripts/run-tests.sh` にサンプルのスクリプト（Go言語用）を用意しています。

### Step 3: CI/CD パイプラインへの組み込み (オプション)
1. 「4-infra-pipeline」で作成した `cloudbuild.yaml` に対して、ビルド・デプロイの*前*にテストを実行するステップを追加します。

```yaml
# 例: cloudbuild.yaml の先頭に追加するステップ
steps:
  - name: 'golang:1.26'
    entrypoint: 'go'
    args: ['test', './...', '-v']
```
