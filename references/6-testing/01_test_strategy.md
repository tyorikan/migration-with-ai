# 01. テスト戦略 (Test Strategy)

Salesforce (SFDC) から Google Cloud (Cloud Run/PostgreSQL) への移行において、品質を担保するための戦略的な考え方を定義します。

## 1. 移行後の品質目標
*   **機能同等性:** 既存の SFDC 上のビジネスロジックが、Google Cloud 上でも同じ結果を返すこと。
*   **データ整合性:** データの移行後、集計値や関連性が SFDC 側と一致していること。
*   **パフォーマンス基準:** 移行後の API レスポンスやバッチ処理時間が、許容範囲内（または向上）であること。

## 2. テストのピラミッド
モダナイズされたアプリでは、以下のバランスでテストを構築します。

1.  **Unit Tests (単体テスト) [大量]:** 
    - ロジックの最小単位を検証。
    - 外部依存（DB等）はモック化。
    - 高速に実行可能。
2.  **Integration Tests (統合テスト) [中量]:**
    - PostgreSQL (Cloud SQL) との連動を検証。
    - コンテナ（Testcontainers 等）を利用して本番に近い環境で実行。
3.  - API エンドポイントや、トリガー発火（Pub/Sub 等）からの一連の流れを確認。

## 3. SFDC 特有の考慮事項
*   **Apex Trigger の移行テスト:** SFDC のトリガーで行っていた複数オブジェクトの同時更新などが、Go/Java のコードでもトランザクションを維持して実行されるか。
*   **ガバナ制限の解消確認:** SFDC で制限されていた大量データ処理が、Google Cloud 上で正しく（タイムアウトせずに）スケールするか。
*   **セキュリティモデル:** SFDC の共有ルール (Sharing Rules) が、アプリケーション側の認可ロジックとして正しく実装されているか。

## 4. データ整合性検証戦略

SFDC から PostgreSQL へのデータ移行後、以下の観点で整合性を担保します。

### 検証レイヤー

| レイヤー | 検証内容 | ツール・手法 | タイミング |
| :--- | :--- | :--- | :--- |
| **レコード件数** | 全テーブルの件数一致 | SQL COUNT + 比較スクリプト | 毎回のロード後 |
| **外部キー整合性** | 孤立レコード（Orphan）= 0 | LEFT JOIN + IS NULL チェック | ロード後 |
| **データ型精度** | Currency, DateTime の精度保持 | Table-driven テスト | CI + カットオーバー前 |
| **文字列変換** | 日本語テキストの往復一致 | ラウンドトリップテスト | CI |
| **Picklist 整合** | 有効値のみ格納されていること | CHECK 制約 or テスト | ロード後 |
| **NULL ハンドリング** | SFDC の空値 → PostgreSQL の NULL | サンプリングテスト | カットオーバー前 |

### サンプルテストコード

具体的なテスト実装は以下のサンプルコードを参照してください:

- [`sample/data_validation_test.go`](sample/data_validation_test.go) — Currency 精度、DateTime タイムゾーン、日本語テキスト、Picklist 値、レコード件数突合、FK 整合性の検証テスト

### カットオーバー判定基準

以下の全条件が満たされた場合のみ、カットオーバーを承認します:

1. ✅ 全テーブルのレコード件数が SFDC エクスポート時の件数と **完全一致**
2. ✅ FK 整合性チェックで孤立レコードが **0 件**
3. ✅ サンプル 100 件のランダムレコードで全項目の値が **完全一致**
4. ✅ 全自動テスト（Unit + Integration + Validation）が **全件パス**

## 5. 検証の自動化
テストコードは、ワークショップを通じて Gemini に生成させますが、その「実行」は CI パイプラインで確実に行います。
- 開発者による PR 作成時に自動実行。
- 失敗した場合はデプロイをブロック（品質ゲート）。
