# 5. ドキュメント作成 (Documentation)

本セクションでは、Gemini を活用して設計書や移行ドキュメント、API仕様書などを自動生成し、メンテナンスする手法を学びます。モダナイゼーション・プロジェクトにおいて、継続的にドキュメントを最新状態に保つためのプラクティスを定着させることが目的です。

## ゴール
1. **Gemini を用いた継続的ドキュメンテーション:** ソースコードやインフラ定義から、Markdown 形式で高品質なドキュメントを生成するフローを体験する。
2. **AI サポートによる設計書・移行ガイド作成:** 既存システムの仕様や移行方針をインプットとし、Gemini に対して適切なプロンプトを与えてベースとなる設計ドキュメントを作成する。
3. **継続的メンテナンス:** ドキュメントをコード化(Docs as Code)し、リポジトリ内で管理することで、システム変更に合わせて容易に更新可能な状態にする。

## ドキュメント生成プロセス

### AI への依頼 (プロンプティング) のベストプラクティス
Gemini にドキュメント生成を依頼する際のポイントです。

1. **コンテキストの提供:** 対象のコード、以前の仕様書、またはアーキテクチャ図（mermaid 形式など）をプロンプトのコンテキストとして含めます。
2. **フォーマットの指定:** 出力形式を明確に指定します（例：「Markdown 形式で出力して」「見出しは H2 で、リスト形式にして」「シーケンス図を mermaid 記法で含めて」）。
3. **対象者の明確化:** 誰向けのドキュメントかを伝えます（例：「経営陣向けのエグゼクティブサマリーとして」「開発者向けの API リファレンスとして」）。
4. **段階的な生成:** 一度に巨大なドキュメントを作成させるのではなく、「まずは目次案を作成して」→「第1章を執筆して」のように段階を踏むと精度が上がります。

## サンプルと実践

このディレクトリには、Gemini を活用して生成されるドキュメントのサンプルテンプレートが格納されています。

*   `docs/architecture/adr-template.md`: アーキテクチャ決定記録 (ADR) のテンプレート。AIにこのフォーマットに沿って記述させることで、一貫性のある意思決定ログを残します。
*   `docs/api/openapi-example.yaml`: (参考) コードなどからGeminiに生成させる API 仕様のサンプル。
*   `tools/generate-docs-prompt.md`: 既存のコードから README などを生成する際に Gemini に与えるプロンプトの例。

## ワークショップでのステップ

### Step 1: 既存コードまたは構成からのドキュメント自動生成
1.  前のセクション (「3-app-modernize」または「4-infra-pipeline」) で作成したコード（例: `main.go` や `main.tf`）を開きます。
2.  Gemini (Cloud Code 拡張機能やチャットパネル、または Gemini for Google Cloud コンソール) に対して、以下のようなプロンプトを実行します。
    *   *「この Go のコードの README.md を Markdown 形式で作成してください。API のエンドポイント一覧と環境変数の説明を含めてください。」*
    *   *「この Terraform コードのインフラ構成図を mermaid 記法で出力し、構成の概要を説明してください。」*

### Step 2: ADR (アーキテクチャ決定記録) の作成
1.  プロジェクトにおける重要な決定（例：「データベースとして Cloud SQL ではなく AlloyDB を選択した理由」）について、Gemini と壁打ちを行います。
2.  壁打ちで得られた情報をもとに、Gemini に ADR のドラフト作成を指示します。
    *   *「先ほどの議論を踏まえて、『データベース エンジンの選定定』に関する ADR (Architecture Decision Record) を `docs/architecture/adr-template.md` のフォーマットで作成してください。」*

### Step 3: プロンプトの共有
1.  チームで効果的だったプロンプトを `tools/prompt-catalog.md` のようなファイルにまとめ、知見としてリポジトリで共有します。

## Docs as Code への統合

生成したドキュメントはソースコードと合わせて本リポジトリで管理します。必要に応じて、Cloud Build などの CI/CD パイプライン内でドキュメント・ジェネレーター（例: MkDocs, Hugo, Docusaurus など）を呼び出し、静的サイトとして社内向けにパブリッシュする仕組みを構築することを推奨します。
