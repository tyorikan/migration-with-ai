# ドキュメント自動生成のためのプロンプトカタログ

このファイルでは、Gemini を使ってコードやインフラ構成からドキュメントを生成する際に、効果的だったプロンプトの例を集めています。チーム内で新しいプロンプトのテクニックを発見した場合は、ここに追加してください。

## 1. README の作成

**ユースケース:** 新しい Go のマイクロサービスを作成したので、その README を作成させたい。

**プロンプト例:**
> 以下のソースコード（`main.go`, `handlers.go`, `go.mod`）を読み込み、このマイクロサービスの `README.md` を作成してください。
>
> 以下のフォーマットと内容を含めること：
> 1.  プロジェクト概要
> 2.  必要な環境と起動方法 (`go run ...` など)
> 3.  API エンドポイントの一覧と各エンドポイントの簡単な説明、リクエスト/レスポンスの例（JSONなどがある場合）
> 4.  環境変数の一覧とその役割（ソースコードから推測して補完すること）
> 5.  出力形式は Markdown で、見出しレベルを適切に使用してください。
>
> --- (これ以下にソースコードを貼り付けるか、Gemini Advanced のファイルアップロード機能を利用)

## 2. Terraform コードからの構成図の生成

**ユースケース:** IaC のリポジトリから、構成の概要図（Mermaid）と説明を作成させたい。

**プロンプト例:**
> 以下の Terraform の設定ファイル（`main.tf`, `variables.tf`, `outputs.tf`）を解析し、インフラ構成のドキュメントを作成してください。
>
> 以下の内容を含めること：
> 1.  作成される主な Google Cloud リソースの概要
> 2.  リソース間の関係性を示すアーキテクチャ構成図（mermaid 形式でコードブロックに出力すること。書式は `mermaid` とすること）
> 3.  入力変数（変数の説明のリスト）
> 4.  出力値の利用方法
>
> --- (これ以下にTFコードを付与)

## 3. 既存コードからの処理フロー (シーケンス図) 生成

**ユースケース:** 複雑な認証フローやデータ処理フローのコードを読み解き、ドキュメント化したい。

**プロンプト例:**
> 以下のファイル群（コントローラー、サービス、リポジトリ層のコード）は、ユーザーのログイン処理とトークン発行のロジックです。
> この処理の流れを、シーケンス図として Markdown 形式のドキュメントにまとめてください。
>
> 以下の指示に従うこと：
> 1.  シーケンス図は mermaid 形式で出力すること。
> 2.  「クライアント」「API ゲートウェイ」「Auth サービス」「ユーザーデータベース」などのアクター（参加者）を適切に定義すること。
> 3.  図の後に、各ステップの補足説明を箇条書きで記載すること。
>
> --- (これ以下にコードを付与)

## 4. ADR (アーキテクチャ決定記録) のドラフト作成

**ユースケース:** ミーティングでの議論（テキスト）や検討事項のメモから、ADR を清書させたい。

**プロンプト例:**
> 以下の議事録メモをもとに、「バッチ処理基盤の選定（Cloud Run Jobs vs Dataflow）」に関する ADR (Architecture Decision Record) のドラフトを、Markdown 形式で作成してください。
>
> 以下のフォーマットに従うこと：
> *   タイトル:
> *   ステータス: (Proposed で)
> *   コンテキストと課題:
> *   決定事項: (今回は Cloud Run Jobs を選択した想定で書くこと)
> *   考慮した選択肢: (Cloud Run Jobs, Dataflow, Cloud Batch などを比較)
> *   決定の理由:
> *   想定される結果 (ポジティブ/ネガティブ):
>
> --- 議事録メモ ---
> (ここに議論のポイントやメモを箇条書きなどで記載)

## 5. テスト仕様書の抽出

**ユースケース:** 既存のテストコード (例: `*_test.go`, `*.spec.ts`) から、どのようなテストケース網羅しているかの仕様書を作りたい。

**プロンプト例:**
> 以下のテストコードから、このモジュールにおいて「何をテストしているのか（テストの意図・仕様）」を抽出し、テスト仕様書としてドキュメント化してください。
>
> 以下の形式で出力すること：
> 1.  テスト対象の関数ごと、またはシナリオごとに見出し (H3) を分ける。
> 2.  各テストケースについては、「入力（前提条件）」「期待される結果」「モックしている外部依存（もしあれば）」を表 (Table) 形式でまとめること。
