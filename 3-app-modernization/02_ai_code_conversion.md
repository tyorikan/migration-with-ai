# 02. AI 駆動でのコード変換

## 目的
SFDC 環境で稼働している Apex コードを、Gemini（Gemini Code Assist または Web UI）を用いて モダンな言語（Go 等）へ変換する具体的な手法とベストプラクティスを学びます。単なる文法変換ではなく、「ステートレスアーキテクチャへの適合」と「依存関係の分離（クリーンアーキテクチャ）」を意識したプロンプトエンジニアリングを体験します。

## 対象となるコード
ワークショップでは `sample/legacy_apex_sample.cls`（Apex クラス） および後述する `legacy_apex_trigger_sample.cls` などを変換対象とします。

## コード変換アプローチ (AI 駆動)

AI にコード変換を依頼する際は、以下のステップを踏むことで品質の高い出力を得られます。

1. **仕様の言語化 (What it does):** AI にソースコードを読み込ませ、何をしているコードなのか、ビジネスロジックを日本語（自然言語）で説明させる。
2. **設計の指示 (How to structure):** ステートレスアーキテクチャや依存性の注入 (DI) など、変換先の設計方針を明示的に指示する。
3. **変換の実行 (Generate code):** 指定した言語 (Go, Python 等) でコードを生成させる。
4. **レビューと修正 (Refine & Test):** 生成されたコードのデータベース接続部分などを、Google Cloud 上で動作するように修正・補完する。

## 実践: パターン別プロンプト集

ワークショップでは、IDE (VS Code + Gemini Code Assist) のチャットウィンドウ、もしくは Google Cloud Console の Gemini ペインに以下のプロンプトを入力して変換を検証します。

### パターン 1: Apex Class (REST/ビジネスロジック) の変換

`legacy_apex_sample.cls` のような標準的なビジネスロジックを変換する際のプロンプトです。

**プロンプト例：**
```
以下の Apex コードを Go 言語 (Go 1.26 以降) を使用してモダンな Web API に変換してください。
変換にあたっては、以下の[要件]に必ず従ってください。

[要件]
- クリーンアーキテクチャの思想を取り入れ、ハンドラー(Handler)、ユースケース(UseCase)、リポジトリ(Repository)として関心を分離してください。
- データベースは Cloud SQL (PostgreSQL) を想定し、ORM は使わずに database/sql モジュールを使用してください。
- Apex の try-catch によるエラーハンドリングは、Go の慣用的な `if err != nil` を使ったエラーハンドリングに変換してください。
- エラー内容はフロントエンドにそのまま返さず、適切な HTTP ステータスコード (400, 500 など) と共に定義されたエラーレスポンス構造体で返却してください。
- テストが容易になるように、外部依存（データベース接続など）はインターフェースとして定義し、DI (Dependency Injection) で注入可能な設計にしてください。
- コードブロック全体を出力し、インポート文を含めてコンパイル可能な状態にしてください。

[Apex コード]
(ここに legacy_apex_sample.cls の内容を貼り付けるか、ファイルを参照)
```

### パターン 2: Apex Trigger のイベント駆動変換

データ変更時に発火する Apex Trigger は、Google Cloud では **Eventarc** や **Pub/Sub** をトリガーとする Cloud Run 関数に変換します。

**プロンプト例：**
```
以下の Apex Trigger のロジックを読み取り、Google Cloud Pub/Sub をトリガーとする Cloud Run ワーカー処理 (Go言語) に変換してください。

[要件]
- SFDC では Account オブジェクトの Update 時に発火していましたが、Google Cloud では Pub/Sub トピックに Account の ID が JSON 形式 `{"accountId": "xxx", "event": "update"}` でパブリッシュされる前提とします。
- Go の標準ライブラリ `net/http` を使用し、Pub/Sub の Push サブスクリプションから HTTP POST 请求を受け取るハンドラーを実装してください。
- 受信したリクエストの Body を JSON デコードし、ペイロード (`message.data`の base64 ペイロード) から accountId を抽出してください。
- ビジネスロジック関数は HTTP ハンドラーとは別の関数に分離し、単体テスト可能にしてください。
- 処理中のエラーはロギングし、HTTP レスポンスとしては（リトライ可能な一時エラーでなければ）200 OK を返して非同期処理を完了させてください。

[Apex Trigger コード]

```

### パターン 3: Batch Apex のバッチ処理変換

大量データ処理を行う Batch Apex は、**Cloud Run Jobs** を想定したスタンドアロンの実行可能ファイルに変換します。

**プロンプト例：**
```
以下の Batch Apex のロジックを読み取り、Google Cloud Run Jobs で実行する Go のバッチアプリケーションに変換してください。

[要件]
- Web サーバー(HTTP)は起動せず、`main()` 実行時に一度だけ処理を行って終了する CLI アプリケーションとして実装してください。
- SFDC では 100 件ずつのバッチチャンクで処理されていましたが、Go では Database から対象レコードを一括で（または Cursor/Offset でチャンク化して）取得するロジックに置き換えてください。
- データベースは Cloud SQL (PostgreSQL) です。
- 処理の進捗や結果は、標準出力 (`log.Printf` など) に出力して、Google Cloud Logging で確認できるようにしてください。
- 環境変数 (DB_HOST, DB_USER, DB_PASS, DB_NAME) から接続情報を取得してください。
```

## 変換されたコードのレビューチェックリスト

AI が生成したコードは必ずレビューし、必要に応じて修正します。以下の観点を確認してください。

- [ ] **DB アクセスの抽象化**: SOQL の代わりに正しい SQL (PostgreSQL ダイアレクト等) または ORM コードに置き換わっているか。
- [ ] **トランザクション管理**: `Savepoint` や `rollback` が、Go の `sql.Tx` などの明確なトランザクション境界に適切にマッピングされているか。
- [ ] **環境変数による設定注入**: 認証情報や DB 接続情報がハードコードされておらず、環境変数 (`os.Getenv` 等) を介して注入される設計か。
- [ ] **ロギングと監視**: 単なる `fmt.Println` ではなく、少なくとも標準の `log` パッケージ、理想的には構造化ロギング (`slog` 等) を使用しているか。
- [ ] **DML 操作の冪等性要件**: もし同じ処理がコンテナのスケールアウトやリトライで2回実行された場合でも問題がないか（UPSERT設計など）。
